# 작업 내역 — 2026-07-15 (Codex 개인 기록)

> **용도:** 내가(Codex) 한 작업의 상세·연대기 기록.
> 동료 Claude에게 넘기는 curated 인계본은 `docs/handoff/codex-handoff.md`에 따로 둔다.

## 세션 개요

사용자는 Claude와 먼저 Furiosa student kit 작업을 진행했고, 이후 Codex가 합류했다.
현재 주제는 Furiosa RNGD용 LLM/VLM 아티팩트 빌드 실습 키트의 스크립트 구조와 코드 리뷰다.

## 한 일 (시간순)

1. **VLM 모델 적합성 질의 대응.**
   - 사용자가 `furiosa-ai/Qwen3-VL-32B-Instruct`가 1개 RNGD NPU(48GB)에서 가능한지 질문.
   - 모델 카드 기준 해당 Furiosa precompiled 모델은 TP 32 PEs = RNGD 4장 구성이어서
     1장 48GB에는 부적합하다고 판단.
   - 1장 목표 후보로 `Qwen/Qwen3-VL-8B-Instruct`, 더 보수적으로 `Qwen/Qwen2.5-VL-7B-Instruct`
     쪽을 추천.

2. **초기 코드 리뷰 수행.**
   - `furiosa-student-kit` repo 기준 working tree가 clean임을 확인.
   - `_kit/scripts/build_artifact.py`, `verify_artifact.py`, `check_resources.sh`와 README 일부를 리뷰.
   - 주요 지적:
     - `build_artifact.py`가 `cpu_per_worker > eff`만 검사하고
       `workers * cpu_per_worker <= eff`는 검사하지 않음.
     - README/가이드 일부 경로가 실제 repo 구조와 맞지 않음.
     - `verify_artifact.py`가 로그 조건을 자동 판정하지 않고 사람이 눈으로 보게 함.

3. **Claude 코드 리뷰 문서 메타 리뷰.**
   - `docs/codereview/2026-07-15-claude-codereview.md`를 검토.
   - Claude의 #1 지적(worker total CPU 미검증)은 타당하나, 제안 수정안은 자동 계산을
     worker-aware하게 바꾸지 않아 반쪽짜리라고 판단.
   - `add_generation_prompt=True`는 타당하지만 핵심 리스크는 아니고, 더 큰 문제는
     `verify_artifact.py`가 자동 검증을 하지 않는 점이라고 정리.
   - 결과 문서 생성:
     - `docs/codereview/codex_2026-07-15-claude-codereview.md`

4. **원본 보존형 Codex 실험 스크립트 작성.**
   - 원본 파일을 건드리지 않고 `codex_` 접두사 파일 생성:
     - `quickstartsmaples/model-preparation/_kit/scripts/codex_build_artifact.py`
     - `quickstartsmaples/model-preparation/_kit/scripts/codex_verify_artifact.py`
   - `codex_build_artifact.py` 개선:
     - `--workers`를 자동 CPU 계산에 반영.
     - `workers * cpu_per_worker <= effective_cpu` 총량 가드 추가.
     - private `ArtifactBuilder` 속성 접근은 명확한 fail-fast 메시지로 감쌈.
   - `codex_verify_artifact.py` 개선:
     - child process로 LLM 실행 로그를 캡처.
     - 금지 경고(`No extend buckets`, `Chunked prefill will be disabled`,
       `Disabling prefix caching`)와 `is_prefix_cache_enabled: true` 존재를 pass/fail로 판정.
     - `add_generation_prompt=True` 사용, 구버전 tokenizer wrapper는 TypeError fallback.
   - 검증:
     - `python -m compileall` 통과.
     - 두 스크립트 `--help` 실행 확인.
     - `codex_build_artifact.py --model dummy --workers 999 --dry-run`으로 CPU 총량 가드 확인.

5. **Claude 모델별 분리 산출물 검토.**
   - 사용자가 `docs/handoff`의 Claude handoff를 보라고 지시.
   - 실제 파일명은 `docs/handoff/claudex-handoff.md`였음. 사용자가 말한
     `claude-handoff.md`와 다름.
   - Claude가 만든 모델별 스크립트 확인:
     - `Qwen3/build_qwen3.py`
     - `Llama/build_llama.py`
     - `Gemma4_try/build_gemma_TEMPLATE.py`
   - 세 파일 `python -m compileall -q` 통과 확인.
   - 판단:
     - Qwen3/Llama 독립 실행본은 사용자 결정("원본 안전 > DRY") 맥락에서는 이해됨.
     - 하지만 최종 학생 키트로는 공용 `build_artifact.py` 1개 유지가 더 낫다고 봄.
     - 이유: 자원 측정/CPU 예산 핵심 로직이 세 벌로 복제되고, 기존 worker 총량 버그도
       세 파일에 그대로 복제됨.
     - Gemma 템플릿은 예외로 유지 가능. 프리셋이 없어서 학생이 `BucketConfig`를 직접
       채우는 실습 목적과 맞음.

6. **협업 기록 작성.**
   - 사용자 지시에 따라 Claude에게 넘기는 인계 파일 생성:
     - `docs/handoff/codex-handoff.md`
   - 이 개인 작업 로그 생성:
     - `history/2026-07-15-codex.md`
