### Qwen3-VL 8B 실행 로그

1. 자원 확인:
   - 명령: `bash quickstartsmaples/model-preparation/_kit/scripts/check_resources.sh`
   - 결과: 실효 CPU 15, `nproc` 128, `/dev/shm` 64MB, RAM 1007GB.
   - 판단: `/dev/shm`가 작으므로 build concurrency는 1 유지.

2. 4096 context dry-run:
   - 명령: `fxb build Qwen/Qwen3-VL-8B-Instruct ./qwen3-vl-8b-instruct-rngd.fxb --dry-run -tp 8 --max-model-len 4096 -O O0`
   - 결과: exit 0.
   - 경고: exact registry entry가 없어 `hidden_size=5120`, `intermediate_size=25600` nearest entry로 fallback.

3. 4096 context actual build:
   - 명령: `fxb build Qwen/Qwen3-VL-8B-Instruct ./qwen3-vl-8b-instruct-rngd.fxb -tp 8 --max-model-len 4096 -O O0 --concurrency 1`
   - 결과: 실패.
   - 오류: `qwen3_vl_32b_w16a16_txt_first_tokenwise`, `tw1024` kernel compile 중 `failed to lower ... no tactic`.

4. 512 context fallback:
   - 명령: `fxb build Qwen/Qwen3-VL-8B-Instruct ./qwen3-vl-8b-instruct-rngd-512.fxb --dry-run -tp 8 --max-model-len 512 -O O0`
   - 결과: dry-run 실패.
   - 오류: `Maximum prefill attention_size (512) must equal maximum decode attention_size (256)`.

5. 256 context fallback:
   - 명령: `fxb build Qwen/Qwen3-VL-8B-Instruct ./qwen3-vl-8b-instruct-rngd-256.fxb --dry-run -tp 8 --max-model-len 256 -O O0`
   - 결과: dry-run 통과.
   - 실제 build 명령: `fxb build Qwen/Qwen3-VL-8B-Instruct ./qwen3-vl-8b-instruct-rngd-256.fxb -tp 8 --max-model-len 256 -O O0 --concurrency 1`
   - 결과: 실패.
   - 오류: `qwen3_vl_32b_w16a16_txt_first_hidden_states_pinned` lowering 실패.

6. tp4 fallback:
   - 명령: `fxb build Qwen/Qwen3-VL-8B-Instruct ./qwen3-vl-8b-instruct-rngd-tp4-256.fxb --dry-run -tp 4 --max-model-len 256 -O O0`
   - 결과: dry-run 통과.
   - 실제 build 명령: `fxb build Qwen/Qwen3-VL-8B-Instruct ./qwen3-vl-8b-instruct-rngd-tp4-256.fxb -tp 4 --max-model-len 256 -O O0 --concurrency 1`
   - 결과: 실패.
   - 오류: 동일한 `txt_first_tokenwise/tw256` lowering 실패.

### 판단

- Qwen3-VL 8B는 로컬 SDK에 architecture 구현이 있지만, 현재 generator registry에는 exact 8B entry가 없다.
- SDK가 Qwen3-VL 32B 계열 generator로 fallback하고, 실제 kernel compile에서 실패한다.
- 따라서 이번 환경에서는 `Qwen/Qwen3-VL-8B-Instruct` FXB 산출물 생성 및 `furiosa-llm serve` 검증까지 도달하지 못했다.
- 실패 원인은 모델 다운로드, host CPU, `/dev/shm`, HBM OOM이 아니라 compiler/generator registry mismatch로 분류한다.

## 추가 작업 — Qwen3-VL 8B custom generator registry 가능성 검토

- Furiosa 공식 문서에서 software stack, model preparation, FXB, supported models를 재확인.
  - compiler/runtime/FXB가 모델 architecture fingerprint와 kernel generation config에 강하게 묶여 있음을 확인.
  - Qwen3-VL architecture 자체는 Text/Image supported이나, 8B Instruct exact validated entry는 문서에 보이지 않음.
- 로컬 SDK 확인.
  - `fxb build --help`에는 사용자 정의 registry 추가 옵션이 없음.
  - `furiosa.kernels.qwen3_vl/common/model_config.py`는 32B config만 로드.
  - naive/optimized wrapper는 모두 `qwen3_vl_32b_w16a16_*` 이름이지만 `hf_config_overrides`로 8B shape를 받을 수 있음.
  - 실제 실패 지점은 optimized text-first kernel의 `hidden_states_pinned` layout hint no-op add로 좁혀짐.
- 판단: registry 추가 방향은 맞지만 공개 CLI/API로 row 하나 추가하는 수준은 아니며, native generator registry 선택과 Python kernel entry가 같이 맞아야 함.
- 다음 실험안: workspace-local overlay로 `furiosa.kernels.qwen3_vl`을 shadowing하고, 8B config/wrapper를 추가한 뒤 optimized text-first path를 naive/generic path로 우회해 최소 bucket actual build부터 재시도.
